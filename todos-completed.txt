PHASE 4: MS GRAPH API CLIENT LAYER [COMPLETED]
================================================================================

GOAL: Type-safe MS Graph API client that can be used from both client-side code
      and SvelteKit server routes. Includes Zod schemas for response validation,
      proper error handling, and rate limit retry logic.

ARCHITECTURE DECISION:
Graph API calls go through SvelteKit server routes (Workers). This keeps the
access token handling server-side after initial acquisition and allows us to
add caching or request coalescing later. The client-side sends the access token
in the Authorization header to our server routes, which forward it to Graph API.

ALTERNATIVE (simpler): Call Graph API directly from the browser using the MSAL
token. This is valid for an MVP and avoids the server route layer. The benefit
of server routes is better error handling, potential caching, and hiding Graph
API details from the frontend.

RECOMMENDED: Start with direct browser-to-Graph calls for simplicity. The MSAL
token is already in the browser. Add server route proxying later if needed.

STEPS:

4.1  Define TypeScript types for Graph entities (src/lib/types/graph.ts):
     - MobileApp: id, displayName, description, publisher, "@odata.type",
       largIcon (nullable), isAssigned, appAvailability
     - MobileAppAssignment: id, intent, target (AssignmentTarget), settings
     - AssignmentTarget: discriminated union on "@odata.type":
       * GroupAssignmentTarget: { groupId }
       * AllDevicesAssignmentTarget: {}
       * AllLicensedUsersAssignmentTarget: {}
       * ExclusionGroupAssignmentTarget: { groupId }
     - AssignmentFilter fields on targets:
       * deviceAndAppManagementAssignmentFilterId (nullable)
       * deviceAndAppManagementAssignmentFilterType: "include" | "exclude" | "none"
     - ConfigurationPolicy: id, name, description, platforms, technologies,
       roleScopeTagIds, settingCount, isAssigned
     - ConfigurationPolicyAssignment: similar to MobileAppAssignment
     - Group: id, displayName, description, groupTypes, membershipRule
     - AssignmentFilter: id, displayName, description, platform, rule
     - GraphPagedResponse<T>: { value: T[], "@odata.nextLink"?: string }
     - BatchRequest: { requests: BatchRequestItem[] }
     - BatchResponse: { responses: BatchResponseItem[] }
     - BatchRequestItem: { id, method, url, body?, headers? }
     - BatchResponseItem: { id, status, body, headers? }

4.2  Create Zod schemas mirroring the types (src/lib/types/schemas.ts):
     - One schema per Graph entity for runtime validation
     - Use z.discriminatedUnion for AssignmentTarget based on "@odata.type"
     - Export both schemas and inferred types (z.infer<typeof schema>)

4.3  Create the Graph API client (src/lib/graph/client.ts):
     - Export class GraphClient or functional module
     - Constructor takes getAccessToken: () => Promise<string>
     - Core method: async request<T>(path, options) that:
       * Gets fresh access token
       * Prepends base URL (https://graph.microsoft.com/beta)
       * Sets Authorization header
       * Sets ConsistencyLevel: eventual header (needed for $search/$count)
       * Calls fetch()
       * Handles HTTP 429: read Retry-After header, wait, retry (max 3 times)
       * Handles HTTP 401: attempt token refresh, retry once
       * Throws typed errors (GraphApiError with status, code, message)
     - Pagination helper: async fetchAll<T>(path, options) that follows
       @odata.nextLink until all pages are fetched (with configurable max pages)
     - Batch helper: async batch(requests: BatchRequestItem[]) that:
       * Chunks requests into groups of 20
       * Sends each chunk as POST /$batch
       * Collects all responses
       * Returns BatchResponseItem[] with individual success/failure per request
       * Handles 429 on individual batch responses (retry failed items)

4.4  Create domain-specific API modules:

     src/lib/graph/apps.ts:
       - listApps(client, options?): paginated app list with $filter, $search support
       - getAppAssignments(client, appId): assignment array
       - assignApp(client, appId, assignments): POST assign with full assignment array

     src/lib/graph/configurations.ts:
       - listConfigPolicies(client, options?)
       - getConfigAssignments(client, policyId)
       - assignConfigPolicy(client, policyId, assignments)

     src/lib/graph/groups.ts:
       - searchGroups(client, query): groups matching display name search
       - listGroups(client, options?): paginated group list

     src/lib/graph/filters.ts:
       - listAssignmentFilters(client): all assignment filters

4.5  Create error types (src/lib/graph/errors.ts):
     - GraphApiError extends Error { status, code, message, requestId }
     - RateLimitError extends GraphApiError { retryAfterSeconds }
     - AuthenticationError extends GraphApiError
     - PermissionError extends GraphApiError
     - toFriendlyMessage(error): user-readable error string

4.6  Create a Graph client store (src/lib/stores/graph.ts):
     - Lazily creates GraphClient instance using auth store's getToken()
     - Exports getGraphClient() function

PHASE 9: BATCH API INTEGRATION FOR BULK OPERATIONS [COMPLETED]
================================================================================

GOAL: Implement the actual execution engine that takes the wizard's output and
      applies assignments efficiently using the MS Graph batch API.

THE CHALLENGE:
Each app/profile's assign endpoint REPLACES all assignments (not append). So for
each item, we must:
  1. GET current assignments
  2. Merge with new assignments (respecting conflict resolution from wizard)
  3. POST the full merged assignment list

For 20 apps, that's 20 GETs + 20 POSTs = 40 requests. The batch API lets us
send 20 per batch, so this is 2 batches.

STEPS:

9.1  Create the assignment execution engine (src/lib/graph/execute.ts):
     - Export async function executeBulkAssignment(params):
       * params: { client, items, groups, intent, filter, conflicts }
       * Returns: { results: AssignmentResult[] }
     - AssignmentResult: { itemId, itemName, status: 'success'|'error', error?: string }

9.2  Implement the merge algorithm (src/lib/graph/merge.ts):
     - mergeAssignments(current, additions, conflicts):
       * current: existing assignments for the app
       * additions: new assignments to add
       * conflicts: user's resolution choices from the wizard
     - For each addition:
       * Check if a current assignment exists for the same group
       * If conflict exists and user chose "skip" → keep current
       * If conflict exists and user chose "update" → replace with new
       * If no conflict → append to the list
     - Handle exclusion groups: add ExclusionGroupAssignmentTarget entries
     - Handle All Devices / All Users: use the special target types
     - Always preserve assignments that aren't part of this bulk operation

9.3  Implement batched execution:
     - Phase A — Fetch current assignments for all items:
       * Build batch: GET /deviceAppManagement/mobileApps/{id}/assignments for each app
       * Send via batch API (groups of 20)
       * Parse responses, map each item to its current assignments
     - Phase B — Compute merged assignments:
       * For each item, run mergeAssignments()
     - Phase C — Apply merged assignments:
       * Build batch: POST .../assign for each app with merged body
       * Send via batch API (groups of 20)
       * Collect results per item
     - Handle mixed app + profile items:
       * Apps use /deviceAppManagement/mobileApps/{id}/assign
       * Profiles use /deviceManagement/configurationPolicies/{id}/assign
       * Can be mixed in the same batch request

9.4  Implement progress reporting:
     - Export progress as a reactive value the UI can subscribe to
     - Track: total items, completed, failed, current phase (fetching/merging/applying)
     - Use a callback or Svelte store pattern:
       ```
       type ProgressCallback = (progress: {
         phase: 'fetching' | 'merging' | 'applying';
         completed: number;
         total: number;
         currentItem?: string;
       }) => void;
       ```

9.5  Implement error handling and retry:
     - If a batch response has 429 for individual items: collect, wait, retry
     - If a batch response has 5xx: retry up to 2 times with backoff
     - If an individual item fails: mark as failed, continue with others
     - Never let one failure block the entire operation
     - Collect all errors with item context for display in StepResults

9.6  Handle rate limiting gracefully:
     - If we get 429 on the batch request itself: wait Retry-After seconds, retry
     - If individual responses within batch are 429: extract those items,
       wait, re-batch just the failed items
     - Show "Rate limited, waiting X seconds..." in the progress UI

PHASE 12: DYNAMIC PERMISSION CONSENT & SCOPE MANAGEMENT [COMPLETED]
================================================================================

GOAL: Build a permission management system that detects which Microsoft Graph
      permissions the current user has consented to, and prompts for incremental
      consent when a feature requires permissions not yet granted. This is the
      foundation for all subsequent phases that add new permission scopes.

PROBLEM THIS SOLVES:
The app currently requests all permissions upfront during sign-in. As we add
new features (devices, Autopilot, etc.) that require new permission scopes,
existing users would need to re-consent. Incremental consent lets us request
only the permissions needed for each feature, when the user first accesses it.
This also means users who only need assignment management don't see scary
permission prompts for device wipe capabilities they won't use.

MSAL INCREMENTAL CONSENT:
MSAL.js supports incremental/dynamic consent. Instead of requesting all scopes
at sign-in, you request additional scopes when needed:
  const tokenResponse = await msalInstance.acquireTokenSilent({
    scopes: ['DeviceManagementManagedDevices.Read.All'],
    account: currentAccount
  });
If the user hasn't consented to the scope, this throws
InteractionRequiredAuthError, and we fall back to acquireTokenPopup() which
shows the consent prompt.

ARCHITECTURE:

  Permission Tiers (mapped to features):
    TIER 1 (Core — requested at sign-in):
      - User.Read
      - DeviceManagementApps.ReadWrite.All
      - DeviceManagementConfiguration.ReadWrite.All
      - Group.Read.All

    TIER 2 (Device Management — requested on first device page visit):
      - DeviceManagementManagedDevices.Read.All

    TIER 3 (Device Actions — requested on first action attempt):
      - DeviceManagementManagedDevices.ReadWrite.All

    TIER 4 (Enrollment/Autopilot — requested on first Autopilot visit):
      - DeviceManagementEnrollment.Read.All

STEPS:

     - Export PermissionTier type: { id, name, description, scopes: string[],
       features: string[] }
     - Export PERMISSION_TIERS constant: array of all tiers with metadata
     - Export FEATURE_PERMISSIONS map: feature route → required scope(s)
       * '/devices' → ['DeviceManagementManagedDevices.Read.All']
       * '/devices/*/actions' → ['DeviceManagementManagedDevices.ReadWrite.All']
